# 脑虫意识与容醒机制

---

## 一、冥想是谁在做的——脑虫意识

进程隔离方案 的 自动记忆整理 是 daemon 进程的后台任务。一个进程在梦中整理记忆。

**Bugs 的冥想不是这样的。** 它不是主Agent 在做，不是某个后台线程在做，而是**脑虫（专门的子Agent）**在做。

```
星空虫族的蜂巢：
    Overmind → 战斗指挥 → 主Agent
    脑虫   → 记忆整理 → 冥想子Agent
    兵虫   → 战斗执行 → 普通子Agent

三者共用一套记忆网格——实时同步。
脑虫改变记忆 → Overmind立即感知。
Overmind学到新东西 → 脑虫立即开始整理。
兵虫发现新事实 → Overmind和脑虫同时接收。
```

### 实现

```rust
// 冥想子Agent：一个特殊的 SubAgent，不执行用户任务，只做记忆维护
pub struct MeditationAgent {
    // 和普通子Agent 一样的基础设施
    // 但它的任务队列里只有冥想类任务
    schedule: CronExpr,           // "0 3 * * *" 凌晨3点
    idle_trigger: Duration,       // 用户空闲 5min 触发
}

// 共用记忆网格——冥想子Agent 直接读写 Memory Mesh
// 所有 Agent（主/子/冥想）看到的记忆是一致的
```

## 二、CapacityWake——事件驱动的容醒

进程隔离方案 的 `capacityWake` 是"满了就睡，有槽就叫醒"。当 bridge 达到容量上限时，新的工作请求不是轮询等待，而是注册一个唤醒信号。一旦有 session 完成，信号触发，bridge 立即醒来接受新工作。

### Bugs 的实现——事件驱动排队

当前调度引擎是轮询模式——检查队列、找一个空闲子Agent、分配。改为事件驱动：

```rust
pub struct CapacityGate {
    available: tokio::sync::Semaphore,   // 信号量 = 空闲槽位数
    waiters: tokio::sync::Notify,         // 等待通知
}

impl CapacityGate {
    // 入队：如果没空位，异步等待（不烧 CPU）
    pub async fn wait_for_slot(&self) -> Slot {
        // 尝试获取许可
        match self.available.try_acquire() {
            Ok(permit) => return Slot { _permit: permit },
            Err(_) => {
                // 满仓 → 休眠等待 → 不轮询烧 CPU
                self.waiters.notified().await;
                // 被唤醒 → 重试
                self.available.acquire().await.unwrap();
            }
        }
    }

    // 任务完成 → 释放 + 唤醒一个等待者
    pub fn release(&self) {
        self.available.add_permits(1);
        self.waiters.notify_one();
    }
}
```

**优势：** 满了不轮询、不烧 CPU。有槽位立即唤醒，延迟 < 1ms。比轮询模式省电、更快。

## 三、每次交互附带记忆——防失忆

人类对话不会每句话都忘记前文。AI Agent 最大的问题就是"失忆"——模型上下文窗口有限，超出就被截断。

Bugs 的方案：**每次交互，根据当前模型的能力，附加一组最相关的记忆到上下文。**

```
用户发来消息
    ↓
查询记忆网格：
    当前场景 | 所有场景 > trust > 知识分类 | 最多 N 条
    ↓
根据模型能力裁剪：
    claude-sonnet-4: N = 20 条（200K 上下文）
    gpt-4o-mini:     N = 10 条（128K 上下文）
    llama3.2:3b:     N = 5 条（32K 上下文）
    ↓
注入 Agent 上下文 → 作为 system prompt 的一部分
    ↓
Agent 回复时自然引用这些记忆
```

### 记忆选择策略

```
选取优先级：
    ① trust > 10 的记忆（高信任优先）
    ② 当前场景优先于全局
    ③ category 多样性——不全是知识类，也要习惯类、规则类
    ④ 最近使用的优先（LRU）
    ⑤ 最多 N 条（由模型上下文窗口决定）

每类至少选 1 条：
    知识类 × 1 → 事实基础
    习惯类 × 1 → 用户偏好
    规则类 × 1 → 行为约束
```

### 防失忆但不保证

| 保证 | 不保证 |
|:----|:----|
| 每次交互都附带相关记忆 | 模型上下文满了会截断 |
| 高信任记忆优先注入 | 低信任记忆可能被省略 |
| 脑虫在后台持续整理 | 上次交互到本次之间新增的记忆可能来不及被纳入 |

> 防失忆是尽力而为，**不是保证。** 就像人记笔记——笔记可以帮你回忆，但不保证你不会忘。

## 四、总结——三者的关系

```
用户交互 → 附带记忆注入 → 主Agent 回复
    ↑                        ↓
    │                   学到新东西 → 写入记忆网格
    │                        ↓
    │                   脑虫（冥想子Agent）→ 持续整理
    │                        ↓
    └──────── 下次交互附带更精准的记忆 ←┘

                      共用 Memory Mesh
              实时同步，所有 Agent 一致
```
