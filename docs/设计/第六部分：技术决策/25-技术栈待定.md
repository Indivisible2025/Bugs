# 技术栈决策

---

## 一、核心语言：Rust vs Go

| 维度 | Rust | Go | 结论 |
|:----|:----|:---|:----|
| 空闲内存 | ~500KB ✅ | ~2MB ⚠️ | Rust |
| 二进制体积 | ~1-5MB ✅ | ~5-10MB ✅ | 平 |
| 调度延迟 | ~100ns ✅ | ~100ns ✅ | 平 |
| 并发模型 | async/tokio | goroutine | 平 |
| GC 抖动 | ❌ 无 ✅ | ⚠️ 有 | Rust |
| 浏览器绑定 | chromiumoxide | chromedp | 平 |
| 嵌入式 KV | redb | Sled 未稳定，redb 为替代 |
| 向量库 | hnsw_rs | 较少选择 | Rust |
| 学习曲线 | 陡 | 平 | Go |
| 生态成熟度 | 中 | 高 | Go |

### 决定

| 决策 | 选择 | 理由 |
|:----|:---:|:----|
| 核心语言 | Rust | 500KB 内存、无 GC 抖动、极致并发 |
| 异步运行时 | Tokio | 工业级验证，work-stealing + LIFO slot |
| HTTP 框架 | Axum | 轻量，基于 Tower 生态 |
| WebSocket | tokio-tungstenite | tokio 原生集成 |
| 序列化 | serde + msgpack | 零拷贝，二进制高效 |

---

## 二、浏览器内核

| 决策 | 选择 | 理由 |
|:----|:---:|:----|
| 默认内核 | Chromium (headless_shell) | 无头模式，CC-BY 兼容 |
| Rust 绑定 | headless_chrome | 成熟 Rust crate，通过 DevTools Protocol 控制 |
| 备选方案 | Firefox geckodriver | 如果 Chromium 体积过大或兼容性问题 |

---

## 三、存储引擎

| 决策 | 选择 | 理由 |
|:----|:---:|:----|
| KV 引擎 | redb (Rust) | 嵌入式，活跃维护，语义化版本 |
| 向量引擎 | hnsw_rs | 纯 Rust，无外部依赖 |
| 会话历史 | JSON 文件 | 简单，人类可读 |

---

## 四、前端技术栈

| 前端 | 技术 | 说明 |
|:----|:----|:----|
| TUI | Ratatui | Rust 原生，内嵌内核，同一进程 |
| Web UI | 原生 HTML + JS + CSS | 内核 HTTP 服务器输出，无框架 |
| GUI 桌面 | egui (Rust) | 原生绘制，非 WebView 套壳，满嘴 WebUI 样式 |
| Android | Kotlin + Compose | 原生 Android，通过 HTTP/WS 连接内核 |
