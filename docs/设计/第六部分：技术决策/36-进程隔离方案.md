# 多用户 + 双进程隔离方案分析

---

## 一、多用户进程隔离

### 方案 A：系统级隔离（推荐——零设计成本）

```
/usr/local/bin/bugs          ← 系统全局二进制（一份）
~user1/.bugs/                ← 用户1 的配置和数据
~user2/.bugs/                ← 用户2 的配置和数据

每人启动自己的 bugs daemon → 独立进程 → 互不干扰
看门狗按用户启动 → systemd --user 服务
```

| 维度 | 说明 |
|:----|:----|
| **是否已支持** | ✅ 当前设计天然支持——每个用户独立 ~/.bugs/ |
| **需要改什么** | ❌ 不需要改。OS 用户隔离就是最干净的隔离 |
| **多用户冲突** | ❌ 没有。不同用户的进程天然隔离 |
| **资源争抢** | ⚠️ 会。两个用户抢同一块 GPU → 需要资源调度 |

**需要补的设计：** 多用户 GPU 资源调度。当用户A 的 Bugs 在跑 LLM 推理时，用户B 的 Bugs 应该被限制或排队。

### 方案 B：Bugs 内多用户（不推荐）

```
同一个 bugs daemon？
    用户A 的 Agent → 用 A 的记忆
    用户B 的 Agent → 用 B 的记忆
```

| 问题 | 说明 |
|:----|:----|
| 崩溃隔离 | ❌ A 的 Agent panic 可能影响 B |
| 安全隔离 | ❌ 脆弱 |
| 记忆隔离 | ✅ 场景隔离可以做到 |
| 复杂度 | ❌ 增加但不带来价值 |

**不推荐。** 系统级隔离（方案A）已经解决了。

## 二、单用户双进程保险

### 要解决的问题

```
当前：1 进程 → 挂了 → 看门狗重启 → 恢复 state.json → 丢最后几秒的操作

想要：2 进程 → 主动挂了 → 备用无痛接管 → 不丢任何状态
```

### 费用分析

| 维度 | 当前单进程 | 双进程热备 |
|:----|:--------|:--------|
| 空闲内存 | < 500KB | ~1MB（翻倍） |
| 部署复杂度 | 单二进制 | 需要管理两个实例 |
| 状态同步 | 无需 | 需要实时同步协议 |
| 写冲突 | 无 | 两个进程同时写磁盘→冲突 |
| 故障检测 | 外部看门狗 | 需要进程间心跳 |
| 恢复延迟 | ~3s（看门狗检测+重启） | ~100ms（热备接管） |

### 判定

| 优点 | 代价 |
|:----|:----|
| 恢复延迟从 3s 降到 100ms | 内存翻倍（破 < 500KB P0） |
| 不丢最后几秒的状态 | 引入状态同步复杂度 |
| 更可靠的用户体验 | 引入写冲突风险 |

**结论：不做双进程热备。** 代价远超收益——多花 500KB 内存只为省 3 秒恢复时间，不符合"极致低资源"的 P0 定位。看门狗 3 秒重启已足够。

## 三、替代方案：快速恢复（已设计）

当前已有的保障足以覆盖绝大多数场景：

| 场景 | 保障 | 恢复延迟 |
|:----|:----|:------:|
| 子Agent panic | catch_unwind | ~0ms |
| OOM 被杀 | 看门狗重启 + state.json | ~3s |
| SIGKILL | 看门狗重启 | ~3s |
| 系统休眠 | 心跳重校准 | ~1s |
| 全进程挂 | 看门狗 + 降级模式 | ~3-10s |

**如果非要追求 < 10ms 级别的恢复：**

不是双进程，而是在**内存中维护两份状态副本**——主 copy 和 shadow copy。主线程操作主 copy，后台 task 实时从主 copy 同步到 shadow。主线程崩了，shadow 还在内存里，立即用 shadow 重建。

```rust
// 不需要第二进程，只需要第二份内存
struct StateShadow<T> {
    primary: Arc<RwLock<T>>,
    shadow: Arc<RwLock<T>>,
}
// 后台 task 每 100ms 同步一次
// primary 崩了 → 用 shadow 立即恢复
```

| vs 双进程 | 内存 | 恢复延迟 | 复杂度 |
|:---------|:---:|:------:|:-----:|
| 双进程 | ~1MB | ~100ms | 高 |
| **影子内存** | **~10KB** | **< 10ms** | **低** |

此方案不破 < 500KB 目标，适合后续补充。

## 四、结论

| 需求 | 方案 | 状态 |
|:----|:----|:---:|
| 多用户隔离 | 系统级 OS 用户隔离 | ✅ 已有 |
| 多用户 GPU 调度 | 需要补设计 | ⏳ 后续 |
| 双进程热备 | **不采纳**（破 P0） | ❌ |
| 快速恢复 | 影子内存 | ⏳ 后续优化 |
| 标准恢复 | 看门狗 + state.json | ✅ 已有 |
