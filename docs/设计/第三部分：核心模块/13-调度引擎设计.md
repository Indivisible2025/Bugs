# 调度引擎设计

> 前置分析 + 详细设计 + 假性能审查。

---

## 一、调度引擎是什么

调度引擎是 Bugs 中**负责任务排队、分发、负载均衡和故障转移的程序组件**。它不是 AI Agent，不调用 LLM，而是纯算法驱动的任务调度器。

类比星空虫族：调度引擎 = 虫后的管理功能——它接收Overmind（主Agent）的指令，创建和管理兵虫（子Agent），决定谁去干什么、谁闲着就派给谁、谁阵亡了就补充谁。

---

## 二、调度引擎做什么

```
用户输入 / 系统事件
    │
    ▼
┌────────────────────────────────────────┐
│          调度引擎 (Scheduler)            │
│                                          │
│  任务排队 ─→ 优先级排序 ─→ 分配 Executor ─→ 执行
│                ↑              │           │
│                │              ▼           │
│            负载均衡       故障转移 ←──────┘
│                            │
│                            ▼
│                        重新分配
└────────────────────────────────────────┘
```

| 职责 | 说明 |
|:----|:----|
| **任务排队** | 接收任务，按优先级/类型入队 |
| **优先级排序** | 高优先级任务先执行 |
| **负载均衡** | 多个 Executor 之间平均分配任务 |
| **故障转移** | 某个 Executor 崩溃，任务自动转给其他 Executor |
| **并发控制** | 限制最大并行数，防止资源耗尽 |
| **任务追踪** | 每个任务的状态：排队中 / 执行中 / 已完成 / 失败 / 超时 |

## 三、AI 调度 vs 算法调度

这是一个关键架构决策：**调度引擎本身要不要用 AI（调 LLM）？**

但首先要把问题问对——问题不是"AI 还是算法"，而是**"笨算法 vs 聪明算法 vs AI"**。

### 笨算法调度

| 特征 | 举例 |
|:----|:----|
| 只看单一维度 | 按优先级排队，同优先级 FIFO |
| 无上下文感知 | 不关心任务类型、场景、历史 |
| 分配策略简单 | 轮询、最少任务、随机 |
| **结果** | **快，但可能做傻事** |

### 聪明算法调度

**不是 AI，但用了更多的上下文和更复杂的策略：**

| 维度 | 笨算法 | 聪明算法 |
|:----|:------|:--------|
| 任务分类 | 只有优先级 | 优先级 + 类型 + 场景 + 所需资源 |
| 分配参考 | 最少任务数 | 多维评分（局部性 + 能力 + 负载 + 亲和） |
| 优先级 | 静态 | 动态（等待时间越长优先级越高 + 防饿死） |
| 场景感知 | 无 | 同一场景的任务倾向分配到同一 Executor |
| 局部性 | 无 | 优先分配给刚完成同类任务的 Executor |

### AI 调度

| 特征 | 说明 |
|:----|:----|
| 原理 | 调度决策由 LLM 做出：判断优先级、分配、是否重试 |
| 优势 | 能理解任务语义，做出上下文感知的决策 |
| 代价 | 每次决策 100ms-5s，需要 GPU，不可预测 |
| 结论 | **不适用于调度热路径** |

### 三层决策模型

```
热路径（每次调度都走，必须微秒级）
├── 任务出队
├── 分配 Executor
├── 状态更新
└── → 纯算法，无 AI，无阻塞

冷路径（可选，仅当配置启用）
├── 动态优先级调整策略
├── 预测性预热（根据用户习惯提前加载）
├── 调度策略自优化
└── → 可用固定规则

人的路径（手动调优）
├── 配置任务优先级
├── 配置场景配额
├── 配置保护路径
└── → 用户手动设置
```

## 四、有哪些方案

### 方案 A：外部消息队列

| 代表 | Celery | Redis Queue | RabbitMQ |
|:----|:------|:-----------|:--------|
| **架构** | Broker + Worker | Broker + Worker | Broker + Worker |
| **延迟** | 亚毫秒 | ~1ms | 亚毫秒 |
| **吞吐** | 百万/分钟 | ~1万/秒 | ~10万/秒 |
| **持久化** | ✅ | 可选 | ✅ |
| **分布** | 多机器 | 多机器 | 多机器 |
| **单任务开销** | ~100μs+ | ~1ms | ~100μs+ |

**不选原因**：需要外部依赖（Redis/RabbitMQ 进程），违背"单二进制"定位，内存开销大，不适合 < 500KB 的目标。

### 方案 B：操作系统线程池

| 代表 | pthread 线程池 | std::thread |
|:----|:-------------|:-----------|
| **每个线程内存** | ~1-8 MB | ~1-8 MB |
| **切换开销** | ~1-10 μs（内核态） | ~1-10 μs |
| **并发能力** | 几千 | 几千 |

**不选原因**：线程太重，1 个线程 1MB 起步，100K 并发需要 100GB 内存。

### 方案 C：协程运行时（推荐方案，二选一）

| 对比项 | Go goroutine | Rust tokio task |
|:------|:------------|:---------------|
| **调度模型** | M:N，work-stealing | M:N，work-stealing |
| **每任务内存** | ~2-4 KB | ~64-200 B |
| **切换开销** | ~200-300 ns（用户态） | ~100-400 ns（用户态） |
| **调度延迟** | ~100-500 ns（同核） | ~100-400 ns（同worker） |
| **单进程并发** | 400-800 万 | 5000 万+ |
| **创建 100 万任务** | ~0.5-2 秒 | ~0.2-1 秒 |
| **是否需 GC** | ✅ 有 GC | ❌ 无 GC |
| **内存占用（空闲）** | ~2 MB | ~500 KB |
| **学习曲线** | 平 | 陡 |

### 方案 D：LMAX Disruptor 模式

| 代表 | Disruptor（Java） | 可类比实现 |
|:----|:-----------------|:----------|
| **原理** | 无锁环形缓冲区 + 内存屏障 | Rust crossbeam / Go channel |
| **延迟** | 亚微秒（sub-μs） | 可参考 |
| **特点** | 避免缓存未命中 + 内核仲裁 | 批量处理、预分配 |
| **适用** | 金融交易、超低延迟场景 | 调度引擎热路径 |

**可作为调度引擎内部队列的参考模式。**

## 五、精品是如何做的

### 参考项目

| 维度 | 做法 |
|:----|:----|
| **模型** | 确定性路由，不是通用调度器 |
| **路由** | 按 channel/peer/account 绑定到特定 agent |
| **子任务** | sessions_spawn 创建子 agent 异步执行 |
| **资源消耗** | TypeScript 运行时，资源占用中等 |

### Go 运行时调度器

| 维度 | 做法 |
|:----|:----|
| **模型** | work-stealing + M:N 线程模型 |
| **本地队列** | 每个 P（逻辑处理器）有自己的 goroutine 队列 |
| **窃取** | P 空闲时从其他 P 窃取 goroutine |
| **栈管理** | 初始 2KB，动态扩缩 |
| **抢占** | 信号式抢占（Go 1.14+） |
| **核心竞争力** | 几百万 goroutine 海量并发 |

### Rust Tokio 调度器

| 维度 | 做法 |
|:----|:----|
| **模型** | work-stealing + LIFO slot |
| **本地队列** | 固定大小 256（无锁环形缓冲区） |
| **LIFO slot** | 被唤醒的任务优先在本核执行，避免跨核 |
| **饥饿保护** | 连续 3 次 LIFO 后改放队列尾部 |
| **窃取节制** | 同时窃取者不超过 worker 数的一半 |
| **全局队列** | Mutex 保护，每 ~10ms 检查一次 |
| **事件检查** | 每 61 个任务后检查 I/O/定时器事件 |
| **核心竞争力** | 微秒级调度，零 GC 抖动 |

## 六、资源占用、性能和效果分析

### 目标 vs 各语言能力

| 指标 | Bugs 目标 | Go goroutine | Rust tokio |
|:----|:---------|:------------|:----------|
| 空闲内存 | < 500 KB | ~2 MB | ✅ ~500 KB |
| 调度延迟 | < 10 μs | ✅ ~100-500 ns | ✅ ~100-400 ns |
| 单节点并发 | **无上限**（资源允许下） | ✅ 几百万 | ✅ 几千万 |
| 调度延迟均稳 | < 10 μs | ⚠️ GC 可能导致暂停 | ✅ 无 GC |
| 单二进制部署 | ✅ 需要静态编译 | ✅ 静态编译 | ✅ 静态编译 |
| 二进制体积 | < 10 MB | ~5-10 MB | ~1-5 MB |
| 开发效率 | — | ✅ 高 | ❌ 低（陡峭） |

### 结论

| 维度 | 推荐 | 理由 |
|:----|:----|:-----|
| **核心引擎** | Rust | 无 GC、500KB 空闲内存、微秒级调度、极致并发 |
| **调度器实现** | Tokio work-stealing | 经过生产验证（数万 Rust 项目），LIFO slot + 工作窃取 |
| **内部队列** | crossbeam / 无锁 MPSC | 低于微秒级的任务传递 |
| **热路径参考** | Disruptor 模式 | 预分配环形缓冲区，零分配热路径 |

> ⚠️ 如果开发效率是优先考量，Go 也是可行选择，但内存占用会从 ~500KB 提升到 ~2MB，且有 GC 抖动风险。

## 七、Bugs 调度引擎要做什么（范围界定）

| 做 | 不做 |
|:--|:----|
| 进程中任务排队和分发 | 跨机器任务调度（那是多实例协作的范畴） |
| Executor 池管理和负载均衡 | 持久化任务队列（非必须，内存即可） |
| 优先级调度 | 分布式任务锁 |
| 故障转移（本机 Executor 间） | 消息可靠性保证（容忍丢任务） |
| 并发控制 | 定时任务/CRON（非核心路径） |
| 任务状态追踪 | Web 管理后台（那是前端的事情） |

## 八、任务结构

每个任务的元数据结构：

```rust
struct Task {
    id: u64,                    // 全局递增 ID
    scene_id: u64,              // 所属场景
    priority: u8,               // 0-255，越低优先级越高
    kind: TaskKind,             // 任务类型
    payload: Vec<u8>,           // 任务内容序列化（Executor 具体干什么）
    created_at: Instant,        // 创建时间戳
    state: TaskState,           // 当前状态
    retry_count: u8,            // 已重试次数
    max_retries: u8,            // 最大重试次数
    timeout: Duration,          // 超时时间
    assigned_to: Option<u64>,   // 分配给哪个 Executor
    tags: Vec<String>,          // 标签，用于路由和过滤
}
```

| 字段 | 说明 |
|:----|:----|
| priority | 0-255，0 最高。不支持负数（对比 trust 用的 -∞ 到 +∞） |
| kind | 推理任务 / 工具调用 / 浏览器请求 / 知识检索 |
| state | pending → queued → running → completed / failed / timed_out |
| retry_count | 每次重试 +1，超过 max_retries 进死信 |
| tags | 如 `["urgent", "scene:project-a"]`，用于高级路由 |

## 九、队列结构

### 方案对比

| 方案 | 复杂度 | 优先级 | 防饿死 | 选否 |
|:----|:-----:|:-----:|:-----:|:---:|
| 简单 FIFO | O(1)/O(1) | ❌ | ✅ 天然公平 | ❌ 无优先级 |
| **多级优先级桶 + 老化** | O(1)/O(k) | ✅ | ✅ **老化机制** | **✅ 选** |
| 优先队列（堆） | O(log n)/O(log n) | ✅ | ❌ 需额外机制 | ❌ 高并发偏慢 |

### 选定：多级优先级桶 + 老化

```
Priority 0      桶 0  最高优先级（紧急任务，用户干预）
Priority 1-3    桶 1  高优先级（交互式任务）
Priority 4-7    桶 2  中优先级（普通任务）
Priority 8-15   桶 3  低优先级（后台任务）
Priority 16-255 桶 4  最低优先级（批量/异步）
```

### 老化机制（防饿死）

任务在队列中等待的时间越长，它的**有效优先级动态提升**：

```
老化规则：
    任务在队列中每等待 T 秒，有效优先级 +1
    如果有效优先级跨越了桶的边界 → 任务自动升桶
    
    等待时间阈值：
    T = 5s（交互式任务快速老化）
    T = 30s（后台任务慢速老化）
    
    正常上限：
    最多升到桶 1（不会进入桶 0 紧急级别，除非手动设置）

    超时逃逸：
    等待总时间超过 max_wait_time（默认 5 分钟）→ 强制升入桶 0
    max_wait_time 可配置
    升入桶 0 后执行一次 → 即使失败也不再重试（防止滥用紧急通道）
```

### 防高优洪水机制

```
每次出队最多连续从桶 0-1 出 N 个任务后，必须从低优先级出 1 个
默认 N = 10
```

## 十一、分配策略

### 方案对比

| 策略 | 特点 | 效果 | 选否 |
|:----|:----|:----|:---:|
| 轮询 (Round-robin) | 无脑轮流 | ❌ 不考虑任何上下文 | ❌ |
| 随机 (Random) | 简单 | ❌ 分布不均匀 | ❌ |
| 最少任务 (Least-loaded) | 只看数量 | ⚠️ 不考虑局部性 | ❌ 单维太傻 |
| **多维评分分配** | 综合多个因素 | ✅ 考虑局部性+能力+负载 | **✅ 选** |

### 选定：多维评分分配

给每个 Executor 打分，选择得分最高的：

```
score = w1 × locality_score
      + w2 × capability_score  
      + w3 × load_score
      + w4 × affinity_score
```

| 维度 | 权重(默认) | 说明 |
|:----|:---------:|:----|
| **局部性 (locality)** | 40% | 刚完成同类任务的 Executor 加分（数据可能在缓存中） |
| **能力匹配 (capability)** | 30% | 已加载所需资源的 Executor 加分（浏览器内核/特定模型） |
| **负载 (load)** | 20% | 当前任务数越少的 Executor 加分 |
| **场景亲和 (affinity)** | 10% | 同一场景的任务倾向分配到同一个 Executor（场景上下文热） |

### 局部性评分

```
最近完成的任务类型与当前任务相同 → locality_score = 1.0
最近完成的任务类型与当前任务同组（如都属于 group:fs）→ 0.7
无相关 → 0.3
```

### 能力匹配评分

```
已加载当前任务所需资源 → capability_score = 1.0（直接锁定）
未加载但可快速加载（如已在内存中的模型）→ 0.6
完全未加载 → 0.2
```

### 负载评分

```
load_score = 1.0 - (current_tasks / max_concurrent)
```

### 场景亲和评分

```
同一场景已有任务在此 Executor 上 → affinity_score = 1.0
不同场景 → 0.5
无历史 → 0.3
```

### 强制绑定（覆盖打分）

```
if task.assigned_to == Some(id) → 直接分配给 id（不经过评分）
if task.kind == BrowserRequest && 有 Executor 已加载浏览器内核 → 优先分配给它
if task.scene_id == scene_id && 有 Executor 有该场景的上下文 → 优先分配给它
```

### 各 TaskKind 的自适应权重

| TaskKind | 局部性% | 能力匹配% | 负载% | 场景亲和% | 理由 |
|:---------|:------:|:--------:|:----:|:--------:|:----|
| 推理任务 | 10 | 50 | 30 | 10 | 模型加载是最大开销 |
| 工具调用 | 30 | 20 | 30 | 20 | 工具资源 + 负载均衡 |
| 文件操作 | 50 | 10 | 30 | 10 | 文件缓存局部性最重要 |
| 浏览器请求 | 20 | 60 | 10 | 10 | 浏览器内核加载是最大开销 |
| 知识检索 | 10 | 20 | 30 | 40 | 场景上下文最重要（RAG 热数据） |
| 默认（未分类） | 30 | 30 | 25 | 15 | 均衡折中 |

权重可通过配置覆盖，每种 TaskKind 独立配置。

## 十二、超时和重试

### 超时层级

```
task.timeout = 任务级超时（由创建者指定，默认 30s）
    ↓（task.timeout > 0）
调度引擎计时
    ↓
到达 timeout → 标记为 timed_out
    ↓
检查 retry_count < max_retries
    ├── 是 → 重新入队，retry_count + 1
    └── 否 → 移入死信队列（dead letter）
```

| 参数 | 默认值 | 说明 |
|:----|:-----:|:----|
| 默认超时 | 30s | 创建任务时不指定超时则用此值 |
| 最大重试次数 | 3 | 超过后进死信 |
| 重试延迟 | 指数退避 | base_delay = 500ms，从 500ms 起步，不是从 0ms 立即重试 |

### 指数退避

```
重试延迟 = base_delay × 2^(retry_count - 1)
base_delay = 500ms
retry=1: 500ms（给系统恢复时间）
retry=2: 1000ms
retry=3: 2000ms
retry=4+: 8000ms（上限）
```

## 十三、并发控制

### 三层限流

```rust
struct ConcurrencyLimits {
    global_max: u32,        // 全局最大并行任务数，默认 100_000
    per_priority_max: [u32; 5],  // 每级优先级桶的最大并行
    per_scene_max: u32,     // 每个场景最大并行，默认 10_000
    per_executor_max: u32,  // 每个 Executor 最大并行，由 Executor 报告
}
```

> 默认值是可配置的安全起点。Bugs 的架构支持**无上限扩展**——global_max 你设为 0 即为"不限制"，你内存多大就跑多大。

### 优先级感知限流

高优先级任务在全局满负载时仍有通道：

```
桶 0（紧急）：硬上限 global_max × 2，超限降为桶 1 排队，不拒绝
桶 1（高优先级）：global_max 满时，最多占用 20% 容量
桶 2（中）：正常
桶 3（低）：global_max 满时，让出容量给高优先级
桶 4（最低）：global_max 满时，强制暂停
```

### 分配检查

```
任务类型是桶 0（紧急）？
    ├── ✅ → 跳过限流检查，直接分配
    └── ❌ →

当前全局任务数 < global_max ?
    ├── ❌ → 当前任务优先级是否属于可抢道级别？
    │       ├── ✅ 且 全局高优任务配额未满 → 分配
    │       └── ❌ → 排队等待
    └── ✅ →
        当前场景任务数 < per_scene_max ?
            ├── ❌ → 排队等待（但定期检查是否可替换低优任务）
            └── ✅ → 分配
```

## 十四、任务状态机

```
        创建
         │
         ▼
      pending ──→ queued ──→ running ──→ completed
                      │          │
                      │          ├──→ failed ──→ 检查重试
                      │          │              ├── 重试 → queued
                      │          │              └── 超限 → dead_letter
                      │          │
                      │          └──→ timed_out ──→ 检查重试
                      │                        ├── 重试 → queued
                      │                        └── 超限 → dead_letter
                      │
                      └── canceled（用户取消）
```

| 状态 | 含义 | 转换条件 |
|:----|:----|:--------|
| pending | 已创建，未入队 | create() |
| queued | 在队列中等待 | enqueue() |
| running | 正在 Executor 上执行 | assign() |
| completed | 正常完成 | Executor 报告 success |
| failed | 执行失败 | Executor 报告 error |
| timed_out | 超时未完成 | 调度引擎检测 |
| dead_letter | 重试用尽或不可恢复 | 超过 max_retries |
| canceled | 用户手动取消 | cancel() |

## 十五、故障转移

### 健康检查

```
调度引擎 → 每 5s 对所有 Executor 发心跳
    ↓
未收到响应 > 15s → 标记 unhealthy
未收到响应 > 30s → 标记 dead
```

### Executor 死亡处理

```
Executor 被标记为 dead
    ↓
遍历该 Executor 所有 running 任务
    ↓
每个任务：
    ├── 重试次数未超 → 重新入队（retry_count + 1）
    └── 重试次数已超 → 移入死信
    ↓
清理 Executor 资源
```

### 全部子Agent 死亡的极端情况

```
所有子Agent 被标记为 dead
    ↓
调度引擎进入"紧急空转"模式：
    ├── 不接受新任务
    ├── 所有 running 任务移入队列（保留重试状态）
    ├── 持续尝试创建新子Agent（每 1s 尝试 1 次）
    ├── 如有任何子Agent 复活/创建 → 立即分配等待中的任务
    └── 超过 5 分钟无复活 → 记录致命错误 → 通知主Agent
```

### 优雅关闭

```
收到 SIGTERM / Ctrl+C
    ↓
调度引擎进入"排空模式"（不接受新任务）
    ↓
等待所有 running 任务完成（最多等 grace_period = 30s）
    ↓
超时未完成的任务 → 标记为 timed_out → 重新入队（下次启动继续）
    ↓
关闭所有 Executor
```

## 十六、并发模型

> 以上所有算法，在极端高并发下（100K+ 任务/秒）必须能跑得动。这一节设计并发模型。

### 16.1 架构：分片 + 工作窃取

```
                    ┌──────────────────────┐
                    │    调度引擎主循环       │
                    │  （协调器，不做热点操作） │
                    └──────┬───────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐
│  分片 0           │ │  分片 1           │ │  分片 N           │
│                   │ │                   │ │                   │
│  本地队列（无锁）   │ │  本地队列（无锁）   │ │  本地队列（无锁）   │
│  Executor 子集    │ │  Executor 子集    │ │  Executor 子集    │
└──────────────────┘ └──────────────────┘ └──────────────────┘
                          │
                          ▼
                   ┌──────────────┐
                   │ 全局溢出队列    │
                   │ (Mutex 保护)   │
                   └──────────────┘
```

**核心思想：没有共享的全局热路径，每个分片独立运行，只在必要时协调。**

### 16.2 分片规则

| 参数 | 默认值 | 说明 |
|:----|:-----:|:----|
| 分片数 | CPU 核心数 | 每个核心1个分片，避免跨核争抢 |
| 每分片本地队列 | 256 | 固定大小环形缓冲区，无锁操作 |
| 每分片 Executor | 总 Executor / 分片数 | 每个分片管理一部分 Executor |

### 16.3 无锁本地队列

每个分片的本地队列使用**固定大小环形缓冲区**（参照 Disruptor 模式）：

```rust
struct LocalQueue<T> {
    buffer: [Cell<T>; 256],   // 固定大小，无动态分配
    head: AtomicU64,           // 生产者索引（仅本分片写入）
    tail: AtomicU64,           // 消费者索引（仅本分片读取）
}

// 单生产者单消费者，无锁
fn push(&self, item: T) -> Result<(), Full> {
    let pos = self.head.fetch_add(1, Relaxed);
    if pos - self.tail.load(Acquire) >= 256 { return Err(Full); }
    self.buffer[pos & 255].set(item);
    Ok(())
}
```

### 16.4 分片间窃取

```
分片 A 空闲（本地队列空）
    ↓ 随机选择一个分片 B
    ↓ 从分片 B 的队尾窃取一半任务（批量，非单个）
    ↓ 放到自己的本地队列
    ↓ 继续执行
```

### 16.5 全局溢出

```
本地队列满 → push 到全局溢出队列（Mutex 保护）
本地队列空且窃取失败 → 从全局溢出队列拉取
```

### 16.6 评分分配（直出，不截断）

多维评分分配：对每个 Executor 做完整四维评分，不提前截断候选。

```
遍历所有 Executor → 对每个 Executor 计算完整四维评分 → 选最高分

假设 1000 个 Executor：
    1000 × 4 次浮点运算 + 1000 次比较 → ~2-3μs
    在 10μs 预算中完全可接受
```

**保险机制：** 当 Executor 数量超过 1000 时（如极端并发场景），评分池限制为最近活跃的前 1000 个 Executor，首次启动的 Executor 优先分配，不参与竞争评分。

**不做两阶段筛选的原因：**
- 节省约 50% 计算量（~1.5μs），但可能排除真正最优的 Executor
- 对于调度效果来说，~1.5μs 不配牺牲决策质量

### 16.7 批量处理

批量处理使用**连续函数**而非离散分段，避免阈值突变：

```
batch_size = max(1, min(128, ceil(log2(current_tasks + 1)) × 8))

典型值：
current_tasks=1     → log2(2)=1    → batch=8
current_tasks=10    → log2(11)≈3.5 → batch=28
current_tasks=100   → log2(101)≈6.7→ batch=54
current_tasks=1000  → log2(1001)≈10→ batch=80
current_tasks=10000 → log2(10001)≈13→ batch=104
current_tasks=100000→ cap          → batch=128
```

连续变化，没有突跳。高负载时自动加大批量以提高吞吐，低负载时自动减小以保证延迟。

## 十八、假性能审查

> 逐个检查每项设计，看是否为了速度牺牲了效果。

### 审查 ①：两阶段分配（16.6）

| 维度 | 定性 |
|:----|:----|
| **问题** | 快速筛选用 load + capability 选 top-3，然后用完整四维评分。**如果最优 Executor 的 load+capability 一般但 locality/affinity 极高，它可能在第一轮就被排除** |
| **后果** | 调度质量下降，选了"差不多"但不是"最好"的 Executor |
| **根因** | 为了省计算量，提前截断了候选集 |
| **判定** | ⚠️ **假性能**：节省了大约 50% 的评分计算量，但可能做出次优分配 |

**修复：** 不提前截断。用**渐进式筛选**代替：
- 先快速计算所有 Executor 的**综合概评分**（一个轻量估算值，O(1) 而非 O(4)）
- 根据概评分选出 top-K（K 可配置，默认 K = 完整候选 = 全部）
- 只在真正分配时才精细化评分
- 或者：仅在 Executor 数量超过阈值（如 100）时才启用两阶段，否则全量评分

**实际成本分析：**
- 每次分配遍历所有 Executor，假设 1000 个 Executor：
  - 全量四维评分：4000 次浮点运算 + 1 次比较 → ~2-3μs
  - 两阶段筛选：2000 次 + 12 次 → ~1-1.5μs
  - 差距 ~1.5μs，在 <10μs 目标内，**不值得为这点性能牺牲效果**

> **结论：不要两阶段筛选，全量评分。**

---

### 审查 ②：固定权重 40/30/20/10（11）

| 维度 | 定性 |
|:----|:----|
| **问题** | `locality 40% + capability 30% + load 20% + affinity 10%` 这组权重的依据是什么？**没有。是我编的** |
| **后果** | 在不需要局部性的场景下（如纯内存检索），locality 权重太高导致错误分配 |
| **根因** | 拍脑袋，没有数据支撑 |
| **判定** | ⚠️ **假性能**：看似合理，实际可能不适合任何现实场景 |

**修复：** 权重不应该是固定的，应该**按任务类型自适应**：

```
推理任务：  locality 10% + capability 50% + load 30% + affinity 10%（模型加载最重要）
文件操作：  locality 50% + capability 10% + load 30% + affinity 10%（文件缓存最重要）
浏览器任务：locality 20% + capability 60% + load 10% + affinity 10%（浏览器内核最重要）
知识检索：  locality 10% + capability 20% + load 30% + affinity 40%（场景上下文最重要）
```

每个 `TaskKind` 有自己的权重配置，既保证了效果，又保持了可预测性。

---

### 审查 ③：立即重试（12）

| 维度 | 定性 |
|:----|:----|
| **问题** | 超时后第一次重试是**立即**执行的（0ms 延迟） |
| **后果** | 如果超时是因为系统过载，立即重试只是**增加负载**，导致更多任务超时 → 雪崩 |
| **根因** | 为了"快速恢复"，没考虑过载场景 |
| **判定** | ⚠️ **假性能**：看似更快恢复，实则可能引发雪崩 |

**修复：** 所有重试都应该有延迟，包括第一次：

```
新策略：
重试延迟 = base_delay × 2^(retry_count - 1)，base_delay = 500ms
retry=1: 500ms
retry=2: 1000ms
retry=3: 2000ms
```

第一次从 500ms 开始，给系统留出恢复时间。

---

### 审查 ④：batch_size 分段跳跃（16.7）

| 维度 | 定性 |
|:----|:----|
| **问题** | `任务 < 100 → batch=1`，`任务 < 10000 → batch=32`，`任务 > 100000 → batch=128`。**在边界处 batch_size 突然变化**，99 个任务时 batch=1（低延迟），100 个任务时 batch=32（低吞吐），延迟突然飙升 32 倍 |
| **后果** | 负载轻微波动就能触发 threshold 跨越，导致延迟剧烈抖动 |
| **根因** | 离散分段，不是连续函数 |
| **判定** | ⚠️ **假性能**：看似自适应，实则引入抖动 |

**修复：** 用连续函数代替离散分段：

```
batch_size = max(1, min(128, log2(current_tasks + 1) × 8))

任务数 1    → batch ≈ 8
任务数 10   → batch ≈ 24
任务数 100  → batch ≈ 48
任务数 1000 → batch ≈ 72
任务数 10000→ batch ≈ 96
任务数 100000→ batch ≈ 128
```

连续变化，没有突跳。

---

### 审查 ⑤：紧急任务不受限（13）

| 维度 | 定性 |
|:----|:----|
| **问题** | 桶 0（紧急任务）跳过 global_max 检查，**理论上无限并发** |
| **后果** | 如果因为代码 Bug 或恶意输入产生了大量紧急任务，系统可能被**紧急任务本身压垮** |
| **根因** | 过于信任"紧急"这个标记，没考虑异常情况 |
| **判定** | ⚠️ **安全漏洞**：不完全是性能问题，但比性能更严重 |

**修复：** 紧急任务也应该有限制，只是限制更宽松：

```
桶 0 硬限制：global_max × 2（紧急任务最多占 2 倍常规容量）
桶 0 单场景限制：per_scene_max × 3
超限 → 排队（优先级降为桶 1），不会拒绝
```

---

### 审查 ⑥：老化上限到桶 1（9）

| 维度 | 定性 |
|:----|:----|
| **问题** | 老化最多升到桶 1，不能进入桶 0（紧急级别） |
| **后果** | 某个桶 4 的任务等了一小时，变成了桶 1，但如果有源源不断的桶 0-1 任务进来，它仍然可能永远轮不到执行 |
| **根因** | 为了保持桶 0 的"纯洁性"，设计得太死 |
| **判定** | ⚠️ **潜在饿死**：极端情况下仍有饿死风险 |

**修复：** 增加超时逃逸机制：

```
等待总时间超过 max_wait_time（默认 5 分钟）→ 强制升入桶 0
max_wait_time 可配置
升入桶 0 后执行一次 → 即使执行失败也不再重试（防止滥用紧急通道）
```

| # | 项 | 判定 | 修复 |
|:-:|:---|:----|:----|
| ① | 两阶段分配 | ⚠️ 假性能 | 全量评分，不截断 |
| ② | 固定权重 | ⚠️ 假性能 | 按任务类型自适应 |
| ③ | 立即重试 | ⚠️ 假性能 | 500ms 起步退避 |
| ④ | batch 跳跃 | ⚠️ 假性能 | 连续函数 |
| ⑤ | 紧急无限制 | ⚠️ 安全漏洞 | 设硬上限 |
| ⑥ | 老化上限桶1 | ⚠️ 潜在饿死 | 超时逃逸到桶 0 |
