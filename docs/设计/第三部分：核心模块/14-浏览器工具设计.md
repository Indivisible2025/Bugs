# 浏览器工具设计

---

## 一、是什么

浏览器工具是 Bugs 的**第一个内置模块**，也是 P1 特性之一。它基于 Chromium 无头模式，通过 DevTools Protocol 控制，为 AI Agent 提供真实的网络搜索和页面渲染能力。Firefox geckodriver 作为备选内核。

**为什么不用 API 搜索？**
- API 搜索（如 Brave/SearchGPT）只能返回文本摘要
- 现代网页大量依赖 JavaScript 渲染，API 拿不到完整内容
- 很多站点（包括技术文档、论坛）需要真实的浏览器环境访问

---

## 二、做什么

```
Agent 请求："搜索 X 主题"
    │
    ▼
┌─────────────────────────────────────┐
│         浏览器管理器 (Browser Mgr)     │
│                                      │
│  请求分发 → 上下文池匹配 → 页面操作 → 内容提取
│                                      │
│  反检测层：指纹随机 / UA伪装 / IP轮换   │
│  稳定层：崩溃重启 / 内存回收费 / 超时处理 │
└─────────────────────────────────────┘
```

## 三、为什么选 Chromium 而非 Firefox

| 维度 | Chromium | Firefox | 结论 |
|:----|:--------|:--------|:---:|
| 控制协议 | DevTools Protocol — 最完善，支持 JS 注入/网络拦截/截图 | Marionette — 接口少，功能不全 | ✅ Chromium |
| Rust 绑定 | `headless_chrome` — 活跃维护 | geckodriver — 需独立进程 | ✅ Chromium |
| headless 体积 | ~100MB | ~150MB | ✅ Chromium |
| 反检测生态 | 指纹/UA/stealth 方案最丰富 | 较少 | ✅ Chromium |
| 隐私 | — | Mozilla 背书 | ✅ Firefox |

> Chromium 是主选，Firefox 作为备选。如果 Chromium 体积/许可有问题就切 Firefox。

## 四、架构——双池模式

```
Browser Manager
    │
    ├── Headless Pool（默认，高并发）
    │   ├── Context #0 → Tab × 8
    │   ├── Context #1 → Tab × 8
    │   └── Context #N → Tab × 8
    │   └── 每个 Context 独立 Chromium headless_shell 子进程
    │
    └── Headful Pool（用户触发，低并发）
        ├── Context #0 → Tab × 2  ← 用户手动处理验证码
        └── Context #1 → Tab × 2  ← 备用窗口
        └── 有头 Chromium 窗口，用户可直接操作
```

| 池 | 并发 | 用途 | 交互方式 |
|:---|:---:|:----|:--------|
| Headless | 无限制 | 自动化搜索、爬取、内容提取 | 仅 API |
| Headful | 1-2 个窗口 | 用户手动处理验证码、调试 | 用户直接操作 |

**两个池互不干扰。** Headless 跑 100 个搜索不阻塞 Headful 窗口。
| Browser Kernel | 实际的浏览器子进程 |

## 五、上下文池

### 池化策略

```
Context 池大小 = min(CPU 核心数 × 2, 16)
每个 Context 最大标签页数 = 8
空闲 Context > 30s → 休眠（释放内存）
闲置 Context > 300s → 销毁
```

**为什么不是每次创建新的？**
- 浏览器子进程启动需要 500ms-2s
- Context 复用避免了冷启动延迟
- 但 Context 也不能无限堆积——每个 Context 约 100-200MB 内存

### 上下文隔离规则

| 隔离维度 | 说明 |
|:--------|:----|
| Cookie | 每个 Context 独立的 Cookie jar |
| 指纹 | 每个 Context 独立的指纹配置（Canvas/WebGL/Audio） |
| 场景关联 | 同一场景的请求尽量分配到同一 Context（享受 Cookie 热数据） |

## 六、反检测机制

### 指纹随机化

每次创建新 Context 时，随机生成浏览器指纹：

| 指纹维度 | 随机范围 |
|:--------|:--------|
| User-Agent | 从真实 UA 库中随机抽取 |
| Canvas 指纹 | 每次渲染加入随机噪声（±1px） |
| WebGL 指纹 | 随机 Hash，模拟不同 GPU 型号 |
| 字体列表 | 真实字体列表，随机排列 |
| 屏幕分辨率 | 常见分辨率集合中随机选择 |
| 语言/时区 | 随 IP 地理位置一致 |

### IP 策略

```
默认：使用本机 IP
可选：通过代理池轮换（用户自行配置）
内置：支持 HTTP/SOCKS5 代理
```

### 验证码兜底

```
页面检测到验证码（含 hCaptcha/reCAPTCHA 特征）
    ↓ 不能绕过 → 报告给 Agent
    ↓ Agent 可选：
    │   ├── 调用 2Captcha/Capsolver 服务（如果配置了API Key）
    │   ├── 切换其他搜索引擎
    │   └── 请求用户手动介入（有头模式下截图）
```

## 七、内容提取

### 提取管线

```
页面加载完成 → 注入 JavaScript 提取脚本 → 执行 → 返回结构化内容
    │
    ├── 正文提取（剔除导航、广告、侧栏、弹窗）
    ├── 代码块保留
    ├── 链接列表提取
    ├── 图片 alt 文本提取
    └── HTML → Markdown 转换
```

### 智能等待

```
固定 timeout（3s） + 智能等待
    ↓
页面加载完成？
    ├── ✅ → 等待额外 0.5s（让延迟 JS 执行完）→ 提取
    ├── ❌ 超时 10s → 提取当前已渲染内容
    └── ❌ 超时 30s → 返回超时错误
```

**不是固定 timeout，而是按页面类型自适应。**

## 八、稳定性保障

| 场景 | 处理 |
|:----|:----|
| 浏览器内核崩溃 | Manager 检测到子进程退出 → 自动重启 → 恢复未完成请求 |
| 内存泄漏 | 每执行 50 次搜索 → 自动回收 Context → 新建 Context |
| 页面卡死 | 单页面超时 30s → 强制关闭该标签页 → 不回收整个 Context |
| 验证码页有头模式 | 用户可随时开启/关闭有头模式（GUI 一键切换） |

---

## 九、有头模式

```
默认：全部走 Headless Pool（生产模式，零 UI 开销）
切换：Headful Pool 独立于 Headless Pool 运行
    ├── 验证码时 → 自动弹出有头窗口 → 用户操作 → 关闭后恢复无头
    ├── 用户手动 → bugs browser head → 打开有头窗口
    └── Headless 任务继续运行不阻塞
```

两个池各自管理生命周期，互不干扰。

## 十、参数汇总

| 参数 | 默认值 | 说明 |
|:----|:-----:|:----|
| Context 池大小 | min(CPU×2, 16) | 最大并行 Context 数 |
| 超时 | 10s | 页面加载超时 |
| Context 回收间隔 | 50 次搜索 | 防止内存泄漏 |
| 闲置休眠 | 30s | 空闲后进入低功耗 |
| 闲置销毁 | 300s | 长期不用则销毁 |

## 十一、假性能自检

| 调度引擎犯的错 | 浏览器工具有没有？ | 说明 |
|:-------------|:---------------|:----|
| 固定权重 | ❌ | 无多维评分 |
| 两阶段截断 | ❌ | 无分配策略 |
| 分段跳跃 | ❌ | Context 策略用 min() 连续函数 |
| 立即重试 | ❌ | 页面超时不自动重试，交给调用方决定 |
| 无上限 | ❌ | 池大小有明确上限 |  
| 随意阈值 | ⚠️ | 50次回收等是经验值，标记为可配置 |
