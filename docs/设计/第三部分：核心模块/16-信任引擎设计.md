# 信任引擎设计

---

## 一、信任引擎是什么

信任引擎是 Bugs 中**管理和计算每条记忆可信度的程序模块**。它不调用 LLM，而是纯数学驱动的评分系统。

核心底层逻辑——**增强和衰减本质上是一种东西**：

```
记忆强度变化 = 增强速率 - 衰减速率

增强速率 > 衰减速率 → 记忆在变强（+X/天）
增强速率 < 衰减速率 → 记忆在变弱（-X/天）
增强速率 = 衰减速率 → 记忆稳定不变
```

---

## 二、信任引擎做什么

```
一条新记忆进入系统
    │
    ▼
┌──────────────────────────────┐
│   信任引擎                     │
│                               │
│ 初始赋值 → 设定增强速率/衰减速率 │
│      ↓                        │
│ 交叉验证 → 增强速率+  ← 事件驱动 │
│ 用户确认 → 增强速率++ ← 最强验证 │
│ 被证伪   → 衰减速率+  ← 加速衰弱 │
│ 时间流逝 → 按净速率自然变化      │
│ 定期询问 → 主动验证 ← 用户把关   │
└──────────────────────────────┘
```

## 三、速率模型（替代离散事件模型）

### 为什么不能用离散事件

| 离散事件模型（旧） | 问题 |
|:-----------------|:----|
| 交叉验证 → trust +2 | 为什么是 2？为什么 3 次验证 = +6 而不是非线性的？ |
| 证伪 → trust -5 | 一条半个月没用过的记忆和天天用的记忆，证伪惩罚一样？不合理 |
| 时间衰减：trust × 0.999 | 衰减只和当前值有关，和历史无关 |

### 速率模型（新）

每条记忆存储两个速率参数，不是单点信任值：

| 参数 | 含义 | 初始值 | 变化方式 |
|:----|:----|:-----:|:--------|
| **enhancement_rate** | 增强速率（+X/天） | 0 | 验证/确认时增加 |
| **decay_rate** | 衰减速率（-X/天） | 0.1/天 | 证伪/久置时增加 |

**当前强度在查询时实时计算：**

```
strength(t) = strength_at_last_update + (enhancement_rate - decay_rate) × days_since_last_update
```

| 场景 | enhancement_rate | decay_rate | 净速率 | 趋势 |
|:----|:---------------:|:----------:|:-----:|:---:|
| 刚创建，未验证 | 0 | 0.1/天 | -0.1/天 | 缓慢衰减 |
| 5 次交叉验证 | +0.5/天 | 0.1/天 | +0.4/天 | 稳步增强 |
| 用户确认过 | +1.0/天 | 0.05/天 | +0.95/天 | 强持续增强 |
| 被证伪 | 0 | +2.0/天 | -2.0/天 | 快速衰减 |
| 长期无人问津 | 0 | +0.3/天 | -0.3/天 | 较慢衰减 |

## 四、事件如何改变速率

### 4.1 交叉验证（增强速率 +）

```
收到新的独立来源验证
    ↓
enhancement_rate += k × reliability
k = 0.1（基础增强量）
reliability = 1.0（默认等权，未来可按来源加权）
```

| 验证次数 | 累计 enhancement_rate | 说明 |
|:------:|:-------------------:|:----|
| 1 次 | +0.1/天 | 刚开始增强 |
| 5 次 | +0.5/天 | 明显正向 |
| 10 次 | +1.0/天 | 强正向 |
| 20 次 | +2.0/天 | 极强正向 |

### 4.2 用户主动确认（增强速率 ++）

```
用户确认"这条记忆还准确"
    ↓
enhancement_rate += 0.5/天（最强验证来源）
decay_rate = max(0.01, decay_rate - 0.1)（同时减少衰减）
```

用户的确认是**最强的验证来源**，因为用户是记忆的最终裁判。

### 4.3 被证伪（衰减速率 +）

```
记忆被新的更高优先级来源明确否定
    ↓
decay_rate += 2.0/天
enhancement_rate = 0（增强归零，不应再增强被否定的事实）
```

### 4.4 自然衰减（知识分类各有初始衰减速率）

| 分类 | 初始 decay_rate | 说明 |
|:----|:--------------:|:----|
| 知识类 | 0.001/天 | 几乎不衰减 |
| 习惯类 | 0.01/天 | 缓慢衰减 |
| 经验类 | 0.05/天 | 较快衰减 |
| 规则类 | 0.001/天 | 几乎不衰减 |
| 关系类 | 0.01/天 | 缓慢衰减 |

## 五、定期主动询问用户验证

定期主动询问用户来判定记忆的准确性。

### 触发机制

```
频率: 可配置，默认每天一次（可选 h/m 级）
触发: 用户空闲时（对话间隙，不打断活跃工作）
选取: 当前场景中 strength 最低的 N 条记忆（N 可配置，默认 3）
```

### 询问流程

```
主Agent → "有一条记忆需要您确认一下："
    │
    ├── "记忆内容：'用户偏好Tab缩进，宽度为4'"
    ├── "来源：子Agent#3 在 project-a 中的观察"
    ├── "上次验证：3天前"
    │
    └── 用户回应：
         ├── "还准确" → enhancement_rate +0.5/天，decay_rate 降低
         ├── "不准确了" → decay_rate +1.0/天，下次可能直接遗忘
         ├── "修正：应为Space缩进" → 更新内容，重置速率
         └── "跳过" → 不改变，下次再问
```

### 不打扰原则

| 规则 | 说明 |
|:----|:----|
| 用户正在对话 | 不弹出，等待空闲 |
| 用户手动关闭 | 本次会话不再询问 |
| 连续 3 次跳过 | 该记忆标记为"不愿意被问"，7天后重试 |
| 频率可调 | 用户可改为"从不询问"/"每小时"/"每天"/"每周" |

## 六、来源可信度与追溯修正

### 为什么需要来源可信度

```
子Agent#3 报告一条记忆
子Agent#5、#7 交叉验证 → 看起来验证充分，增强速率很高
三个月后：用户发现这条记忆是错的（当时情况特殊）
    ↓
现在的问题：不仅要否定这条记忆，
还要追溯到子Agent#5 和 #7 的那两次无效投票
```

### 来源可信度模型

每个来源（Agent）有自己的 credibility，初始 1.0：

```
当来源验证的记忆被用户确认：
    credibility = min(1.0, credibility × 1.05)

当来源验证的记忆被证伪：
    credibility = credibility × 0.8
    这条"错误投票"记录在来源历史中

当来源长期无任何验证记录（>7天）：
    credibility = 0.5 + (credibility - 0.5) × 0.9/天
    缓慢回归 0.5（"不确定"状态）
```

### credibility 对验证权重的影响

```
交叉验证的实际增强量：
    Δ_enhancement = k × credibility × base_k

k = 0.1（基础值）
credibility = 1.0 → Δ = 0.1
credibility = 0.5 → Δ = 0.05
credibility = 0.1 → Δ = 0.01
```

### 追溯修正机制

```
记忆被证伪时：
    ↓
① 记忆自身：decay_rate +2.0/天，enhancement_rate = 0
② 来源追溯：找出所有验证过此记忆的来源
    ├── 对每个来源：credibility × 0.8
    ├── 记录："Agent#X 对记忆#Y 的验证被推翻"
    └── 如果同一来源累计 3 次错误验证：
        └── 标记为"需要人工审核"
③ 不需要回滚所有受影响记忆（太昂贵，会连锁反应）
    但新查询计算 strength 时，credibility 已经降低了
    → 之前靠这个来源验证的记忆，实际 strength 比表面数字低
```

## 七、记忆合并的加权算法

### 问题

```
记忆A："用户偏好Tab缩进"    enhancement_rate: +0.5/天, source_count: 5
记忆B："用户缩进宽度为4"     enhancement_rate: +0.1/天, source_count: 1
    ↓
cosine_sim > 阈值 → 合并为："用户偏好Tab缩进，宽度为4"

合并后的速率不能简单取平均值
```

### 合并规则

```
检查重复来源：
    ├── source_A ∩ source_B 有重复：
    │   └── 合并后的 source_count = |union(sources_A, sources_B)|
    │       不是 |A| + |B|（去重）
    │       同一 Agent 在 A 和 B 中都出现 ≠ 交叉验证
    
合并增强速率：
    △) 重叠内容（两条记忆都涉及的语义区域）→
        new_rate = (rate_A × source_count_A + rate_B × source_count_B) / (source_count_A + source_count_B)
        按独立来源数加权，而不是简单算术平均
    
    ② A 增有内容 → 保持 A 的增强速率不变
    ③ B 独有内容 → 保持 B 的增强速率不变
    
合并衰减速率：
    new_decay = min(decay_A, decay_B)
    取更稳定的衰减速率（合并后的记忆不应比合并前更不稳定）
```

### 相似度与合并深度的关系

```
cosine_sim > 0.95: 高度重叠，几乎完全合并
cosine_sim 0.85-0.95: 中等重叠，按上述规则合并
cosine_sim 0.75-0.85: 低重叠，不合并但建立"关联"指针
cosine_sim < 0.75: 视为不同记忆，不处理
```

## 八、记忆部分更新的信用继承

### 问题

```
记忆："模块A依赖模块B、模块C、模块D"
用户修正："模块A依赖模块B、模块C、模块E"
    ↓
模块B和C 的依赖关系没变 → 这部分不该从零开始
只有 D→E 需要重新评估
```

### 信用继承规则

```
更新记忆时，计算新旧内容的余弦相似度：
    
    sim > 0.9（微小修改，如参数调整）:
        inherited_rate = old_enhancement_rate × 0.9
        new_enhancement_rate = inherited_rate + 0.05
        保留大部分信用
        
    sim 0.6-0.9（中等修改，如部分改写）:
        inherited_rate = old_enhancement_rate × 0.5
        new_enhancement_rate = inherited_rate + 0.05
        保留一半信用
        
    sim < 0.6（大幅改动，视为新记忆）:
        new_enhancement_rate = 0.05
        不继承信用
    
    旧记忆不删除 → 作为历史版本保留，但标记为"已取代"
    如果新记忆后来也被证伪 → 可以回退到旧版本
```

## 九、记忆标签（扩展版）

每条记忆的多维标签：

```
{
  agent_id: <首次报告的Agent ID>,
  scene_id: <记忆所属场景>,
  scope: "single"         // 仅此场景
       | "all"            // 所有场景
       | ["s1", "s2"],    // 指定多个场景
  owners: ["MainAgent", "SubAgent#1", "SubAgent#3"],  // 拥有此记忆的Agent
  category: "knowledge" | "habit" | "experience" | "rule" | "relation",
  subagent_type: "browser" | "file_op" | "reasoning" | "general",
  enhancement_rate: 0.5,   // +0.5/天
  decay_rate: 0.1,          // -0.1/天
  strength_cached: 8.3,     // 上次计算的强度（实时计算，仅作缓存）
  last_updated: <timestamp>,
  last_validated: <timestamp>,
  validation_count: 5,
  source_count: 3,          // 独立来源数
  seq: 3
}
```

### scope 说明

| scope | 含义 | 示例 |
|:------|:----|:----|
| "single" | 仅当前场景有效，换场景不可见 | project-a 的 API 约定 |
| "all" | 所有场景有效，全局可见 | 用户偏好 Tab 缩进 |
| ["s1","s2"] | 指定场景列表可见 | project-a 和 project-b 共用的库约定 |

### owners 说明

| owners | 含义 |
|:-------|:----|
| ["MainAgent"] | 仅主Agent 可见 |
| ["MainAgent", "SubAgent#3"] | 主Agent + 特定子Agent |
| ["SubAgent:file_op"] | 所有文件操作类子Agent 可见 |
| ["AllAgents"] | 所有Agent 可见 |

---

## 十、标签格式在存储中的使用

```
检索时过滤条件：
    owner = 当前Agent OR "AllAgents"
    AND scope = "all" OR scope 包含当前场景
    AND subagent_type = 当前Agent 类型 OR "general"
排序：strength 降序
```

## 十二、全功能性能特性

信任引擎的全功能（增强/衰减、证伪追溯、合并加权、部分更新继承、来源可信度、定期验证）从 v1.0 起完整实现。

### 每操作计算成本

| 操作 | 100 万条记忆批量成本 | 扩容特性 |
|:----|:-------------------|:--------|
| 交叉验证（含 credibility 查表） | 微秒级 O(1) | 无影响 |
| 证伪追溯（含来源可信度罚） | 微秒级（投票来源通常 1-10 条） | 无影响 |
| 相似度去重（向量内积 + hnsw 检索） | ~0.1ms O(768+D×log N) | hnsw 对数增长 |
| 合并加权（含独立来源去重） | 微秒级（候选通常 2-5 条） | 无影响 |
| 部分更新继承（余弦相似度） | ~0.01ms O(768) | 无影响 |
| 定时衰减扫描 | ~50ms/10万条（异步，不阻塞） | 线性增长 |

### 大规模自适应

随着记忆规模增长，系统自动调整执行频率，不关闭功能：

| 记忆数 | 调整 |
|:------|:----|
| < 10 万条 | 全功能，每次写入全量检查 |
| 10-50 万条 | 去重检测从全量精确匹配降为 top-K 候选匹配 |
| 50 万+条 | 衰减扫描间隔从每天一次延长到每周一次 |
| 100 万+条 | 启用分片索引，并行处理 |
