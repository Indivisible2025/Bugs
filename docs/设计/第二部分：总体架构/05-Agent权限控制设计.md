# Agent 权限控制设计

> 基于竞品深度分析（33-竞品深度分析.md）的结论。

---

## 一、为什么需要 Agent 权限控制

| 问题 | 后果 | 没有权限控制会怎样 |
|:----|:----|:----------------|
| Agent 调了不应该调的工具 | 意外操作 | ❌ 可能删文件、执行危险命令 |
| Agent 在错误场景用了敏感能力 | 信息泄露 | ❌ 项目A的记忆被项目B看到 |
| 第三方模块越权 | 系统被入侵 | ❌ 模块可以做任何事 |
| 子Agent 不受控执行危险操作 | 安全失控 | ❌ 主Agent 派遣的子Agent 可能被恶意注入 |
| 每个操作都弹审批 | 用户体验差 | ❌ 频繁确认，无法自动化 |

---

## 二、权限体系层级

```
① 全局默认权限（~/.bugs/config.json 中的 tools.profile）
    ↓  基础白名单，适用于所有场景
② 场景覆盖权限（scene 的独立配置）
    ↓  场景内可放宽或收紧
③ Executor 运行时权限（单次任务临时调整）
    ↓  不持久化，用完即弃
④ 信任引擎关联（自动审批的阈值）
    ↓  信任值高的 Agent 有更多自主权
⑤ 保护路径（永不自动放行）
    ↓  系统关键路径，永远需要确认
```

## 三、审批级别 + 审批选项

### 6 级审批模式（不变）

| 级别 | 说明 | 对应场景 |
|:----|:----|:--------|
| **全部审批** | 每个操作都弹窗确认 | 新人 / 不信任场景 |
| **信任审批** | 信任值足够则自动放行 | 日常编码 |
| **计划模式** | 只读不写，输出计划再确认 | 大型重构前 |
| **自动执行** | 信任引擎 + 安全规则双重把关 | 自动化流水线 |
| **仅白名单** | 只放行明确许可的操作 | CI/CD |
| **全部放行** | 无任何检查 | 容器/沙箱内 |

### 每次审批弹窗的三态选项

```
┌────────────────────────────────────────────┐
│  操作: 读取文件                              │
│  Agent: 子Agent#3（代表主Agent派遣）         │
│  目标: /home/user/project/src/main.rs       │
│  目的: 查找函数签名以确定API调用参数           │
│  风险: 仅读取，不会修改文件                   │
│                                              │
│  [永久允许]  [允许一次]  [不允许]              │
└────────────────────────────────────────────┘
```

| 选项 | 含义 | 效果 |
|:----|:----|:----|
| **永久允许** | 写入场景记忆（trust +1），以后同类操作自动放行 | 持久化 |
| **允许一次** | 本次允许，下次再问 | 仅本次有效 |
| **不允许** | 本次拒绝，不写入记忆 | 即时生效 |

### 弹窗语言要求

| 要求 | 说明 |
|:----|:----|
| **系统默认语言** | 弹窗文案跟随系统 locale（中文/英文/日文等） |
| **操作描述** | 具体到文件名、命令名、URL，不模糊 |
| **目的说明** | 为什么需要这个操作（由主Agent 在派遣子Agent 时声明） |
| **风险评估** | 只读/会修改/会联网/会执行代码，一目了然 |

## 四、工具权限矩阵

| 工具 | 默认 | 可禁用 | 可调节 |
|:----|:----:|:------:|:------:|
| 文件读写 | ✅ 允许 | ✅ | 可限制到特定路径 |
| Shell 执行 | ✅ 允许 | ✅ | 可限制到特定命令 |
| 记忆操作 | ✅ 允许 | ✅ | 可限制读写范围 |
| 知识检索 | ✅ 允许 | ✅ | 可限制数据源 |
| 浏览器 | ❌ 默认禁用 | ✅ | 启用后可限制域名 |
| 代码沙箱 | ❌ 默认禁用 | ✅ | 启用后可限制资源 |
| 模块工具 | 取决于模块自身安全声明 | ✅ | 可按模块 ID 黑白名单 |

## 五、保护路径

以下路径在**任何审批级别下**（bypassPermissions 除外）都不会自动放行：

| 路径 | 原因 |
|:----|:----|
| `~/.bugs/` | Bugs 自身配置，修改需确认 |
| `~/.ssh/` | SSH 密钥，安全敏感 |
| `/etc/` 下系统配置 | 系统安全 |
| `~/.bashrc` / `~/.zshrc` / `~/.profile` | Shell 配置，可能被注入恶意命令 |
| `.git/` 目录 | 仓库结构保护 |
| `.gitconfig` | Git 配置保护 |
| 用户通过配置声明的路径 | 自定义保护 |

## 六、权限与信任引擎的关联

```
信任值 < 0  → 只能使用"全部审批"模式，每次操作都弹确认
信任值 ≥ 0  → 默认模式（计划+审批）
信任值 ≥ trust_auto_threshold → 信任审批（文件操作自动放行）
信任值 ≥ trust_full_auto_threshold → 自动执行模式
信任值 < trust_force_restrict_threshold → 强制降级为仅白名单模式
```

默认值：

| 参数 | 默认 | 含义 |
|:----|:---:|:----|
| trust_auto_threshold | 10 | 超过此值自动放行文件操作 |
| trust_full_auto_threshold | 30 | 超过此值自动执行 |
| trust_force_restrict_threshold | -10 | 低于此值强制白名单 |

> 这些阈值不是固定不变的死数，是**可配置的初始默认值**。每个场景、每个用户可以根据信任引擎的实际表现调整。阈值之间的间隔（20）保证了从"自动放行"到"完全自动"有一段缓冲期，避免因信任值轻微波动在两种模式间反复横跳。

> 信任引擎详细设计见 **16-信任引擎设计.md**。

## 七、配置结构（草案）

```json
{
  "tools": {
    "profile": "coding",
    "allow": ["group:fs", "group:runtime", "group:memory", "group:knowledge"],
    "deny": ["group:browser", "group:sandbox"],
    "protectionPaths": ["~/.bugs/", "/etc/"],
    "byAgent": {
      "scene-project-a": {
        "allow": ["group:browser"]
      }
    }
  }
}
```

## 八、子Agent 权限继承

子Agent 由主Agent 派遣，默认继承主Agent 的权限集，但不能越权：

```
主Agent 权限 = 全局配置 ∪ 场景配置
子Agent 权限 = 主Agent 权限 ∩ 子Agent 安全限制（只能缩小，不能扩大）
```

| 主Agent 有权限 | 子Agent 继承 | 原因 |
|:-------------|:-----------|:----|
| 文件读写 | ✅ 可读写 | 执行任务的刚需 |
| 执行 Shell | ⚠️ 可执行但路径受限 | 限制到工作目录 |
| 网络请求 | ⚠️ 可请求但域名受限 | 限制到任务声明的域名 |
| 记忆读写 | ✅ 可读写（仅当前场景） | 子Agent 不需要跨场景记忆 |
| 浏览器 | ⚠️ 主Agent 有则子Agent 也有 | 不建议子Agent 频繁使用（资源重） |

主Agent 可以在派遣子Agent 时额外收紧权限。
