# 竞品深度分析

> 基于对 OpenClaw、进程隔离方案、OpenCode/Crush、MCP 四个系统的实际调研。

---

## 一、工具体系对比

### OpenClaw

| 维度 | 详情 |
|:----|:------|
| **工具分类** | 10 个分组：runtime / fs / web / browser / sessions / memory / automation / messaging / nodes / media |
| **工具配置** | 4 级 profile（minimal / coding / messaging / full） |
| **权限控制** | allow/deny 列表（deny 优先）、byProvider（按模型限制）、toolsBySender（按用户限制） |
| **审批机制** | elevated exec（沙箱外执行需特殊授权）、exec approvals |
| **工具组** | `group:fs` = read+write+edit+apply_patch 等批量控制 |
| **特殊机制** | 工具循环检测（loopDetection）、沙箱工具策略分层 |

### 进程隔离方案

| 维度 | 详情 |
|:----|:------|
| **工具分类** | 文件操作 / Shell / Web / Agent / 子Agent 等 |
| **权限控制** | 6 级模式：default / acceptEdits / plan / auto / dontAsk / bypassPermissions |
| **审批机制** | 自动审批范围（文件编辑+常见文件系统命令）、保护路径（永不自动审批） |
| **自动模式** | 独立分类器模型审查每个操作，阻止越权行为 |
| **dontAsk** | 只执行白名单工具，其他全部拒绝，用于 CI |

### OpenCode / Crush

| 维度 | 详情 |
|:----|:------|
| **工具分类** | 文件操作 / Shell / 搜索 / 网络 / 子Agent |
| **权限控制** | 每次调用弹权限对话框（allow/deny/session-allow） |
| **审批机制** | 会话级授权，无模型级/用户级细分 |

### 关键发现

| 对比项 | OpenClaw | 进程隔离方案 | OpenCode |
|:------|:--------:|:-----------:|:--------:|
| 工具分组 | ✅ 10组 | ❌ 无显式分组 | ❌ 无分组 |
| 多级 profile | ✅ 4级 | ❌ 6级模式（不同概念） | ❌ |
| 按模型限制工具 | ✅ | ❌ | ❌ |
| 按用户限制工具 | ✅ | ✅ managed settings | ❌ |
| 工具循环检测 | ✅ | ❌ | ❌ |
| 沙箱隔离 | ✅ | ✅ | ❌ |
| 自动审批模式 | ✅（elevated） | ✅（acceptEdits） | ❌ |

## 二、权限体系对比

### OpenClaw 权限层级

```
工具 profile（基础白名单）
    ↓
byProvider（按模型/提供商限制）
    ↓
allow/deny（全局）
    ↓
toolsBySender（按用户限制）  ← deny 优先
    ↓
沙箱策略（sandbox模式）
    ↓
elevated exec（特权执行）
```

### 进程隔离方案 权限层级

```
permission mode（6种模式）
    ↓
allow/ask/deny rules（自定义规则）
    ↓
protected paths（保护路径，永不自动审批）
    ↓
auto mode classifier（自动模式下的安全分类器）
    ↓
managed settings（企业托管配置）
```

### 对 Bugs 的启示

| 启示 | 来源 | Bugs 怎么做 |
|:----|:----|:-----------|
| 工具应有分组 | OpenClaw | 必选/可选/插件三级 + 功能分组 |
| 权限应有层级 | 两者 | 全局→场景→单次三级覆盖 |
| 应有保护路径 | 进程隔离方案 | 系统关键路径永不自动放行 |
| 模型差异影响工具可见性 | OpenClaw | 不同模型能力不同，工具应随之调整 |
| 免审批模式需要安全兜底 | 进程隔离方案 auto mode | 引入信任机制作为自动审批的基础 |

## 四、三类扩展的明确定义

基于实际调研，修正之前的分类：

| 类型 | 别名 | 本质 | 形式 | 典型文件 |
|:----|:----|:----|:----|:--------|
| **插件** | Plugin | 运行时扩展，添加新能力 | 代码模块（编译或脚本） | openclaw.plugin.json |
| **技能** | Skill | 指令扩展，教 Agent 用法 | 纯 Markdown 指令 | SKILL.md |
| **协议** | MCP | 标准协议，对接外部服务 | JSON-RPC 2.0 端点 | 无固定文件 |

Bugs 三者都应支持。

## 五、工具分类建议

基于竞品分析，Bugs 的工具应按两个维度分类：

### 维度一：来源层级

| 来源 | 说明 |
|:----|:----|
| 必选工具 | 核心能力，内置不可卸载（文件/Shell/记忆操作/知识检索） |
| 可选工具 | 内置但可按需启用（浏览器/代码沙箱/API调用/文件解析） |
| 插件工具 | 第三方提供，安装后可用 |
| MCP 工具 | 通过 MCP 协议对接的外部工具 |

### 维度二：功能分组

| 分组 | 工具 | 必选/可选 |
|:----|:----|:--------:|
| group:fs | 文件读写、目录操作 | 必选 |
| group:runtime | Shell 执行、进程管理 | 必选 |
| group:memory | 记忆检索、记忆写入 | 必选 |
| group:knowledge | RAG 查询、知识库操作 | 必选 |
| group:browser | 网页浏览、内容提取 | 可选 |
| group:sandbox | 代码沙箱执行 | 可选 |
| group:network | HTTP 请求、API 调用 | 可选 |
| group:parse | 文件解析（PDF/Word/Excel） | 可选 |

## 六、Agent 权限体系建议

基于竞品分析，Bugs 的权限体系应为：

```
全局默认权限（tools.profile）
    ↓
场景覆盖权限（按场景配置）
    ↓
Executor 运行时权限（按具体任务临时放宽/收紧）
    ↓
信任引擎关联（信任值高的自动化审批阈值低）
    ↓
保护路径（系统关键路径永远需要确认）
```

### 审批级别

| 级别 | 说明 | 对应 进程隔离方案 |
|:----|:----|:----------------|
| 全部审批 | 每个操作都确认 | default |
| 信任审批 | 信任值足够的自动放行 | acceptEdits |
| 计划模式 | 只读不写，出计划再确认 | plan |
| 自动执行 | 信任引擎 + 安全分类器双重把关 | auto |
| 仅白名单 | 只放行明确许可的操作 | dontAsk |
| 全部放行 | 无任何检查，仅容器内使用 | bypassPermissions |
