# 安全设计

---

## 一、认证

### 本地访客
```
来源 127.0.0.1 或 Unix Socket → 自动信任
无 Token 要求
```

### 远程访客
```
Token 认证（Bearer Token）
首次启动 → 自动生成 256-bit 随机 Token → 存储于 ~/.bugs/auth.json
用户可手动替换 Token
Token 泄露 → 用户手动重新生成 → bugsctl token regenerate
```

### 可选：mTLS
```
远程域名绑定时可启用双向 TLS
需要用户提供 CA 证书和服务端证书
适用场景：企业部署
```

## 六、韧性——进程崩溃恢复（协程模型的补救）

Bugs 是单进程协程模型——进程一挂全挂。必须用外挂式保障补这个缺陷。

### 6.1 崩溃分类与对策

| 崩溃类型 | 示例 | 检测方式 | 恢复策略 |
|:--------|:----|:--------|:--------|
| **子Agent panic** | Rust unwrap 失败 | catch_unwind 捕获 | 标记任务失败，不影响进程 |
| **进程 OOM 被杀** | OS 内存不足 → SIGKILL | 看门狗检测进程消失 | 等待系统资源恢复 → 重启进程 → state.json 恢复 |
| **SIGTERM（用户关闭）** | bugs stop / systemd restart | 信号处理 | 优雅关闭：排空任务 → 刷新 state.json → 退出 |
| **SIGKILL（强制杀）** | kill -9 / OOM | 无法捕获 | 看门狗重启 → state.json 恢复（容忍丢最后的操作） |
| **磁盘满** | 写入失败 | 每次写前检查可用空间 | 降级为内存模式 → 告警 → 等待空间恢复 |
| **网络断** | LLM API 超时 | 请求超时 | 重试指数退避 → 降级到本地 Ollama → 缓存响应 |
| **死循环/hang** | 子Agent 无限循环 | TaskBudget CPU 超限 | 强制中止 task → 回收资源 |
| **系统休眠恢复** | 笔记本合盖打开 | 定时器跳变 | 检测时间跳变 → 心跳重校准 → 重连 LLM |
| **系统资源不足** | CPU/内存被其他进程占满 | 定期系统指标采样 | 压缩内存 → 暂停非关键任务 → 等待资源恢复后自动扩容 |

### 6.2 看门狗扩展

```
systemd / supervisor 保活层
    │
    ▼
┌─────────────────────────────────┐
│  看门狗（增强版）                 │
│                                  │
│  每 5s：检查 bugs 进程存活        │
│  每 10s：检查 /api/health 响应    │
│  每 30s：检查系统资源            │
│        ├── 内存可用 < 10% → 触发压缩 │
│        ├── 磁盘可用 < 100MB → 告警   │
│        └── CPU 持续 100% → 限流      │
│                                  │
│  进程死亡 → 等 3s → 重启           │
│  重启失败 → 指数退避（3s/9s/27s）   │
│  连续 3 次失败 → 降级模式启动      │
└─────────────────────────────────┘
```

### 6.3 降级模式

当系统资源不足且无法立即恢复时，Bugs 自动降级：

```
正常模式：
    全功能运行（浏览器/记忆索引/全量日志）

资源预警（内存 < 20%）：
    释放浏览器 Headless Context → 仅保留 Headful
    向量索引压缩（量化维度从 768 降到 256）
    衰减扫描暂停

紧急模式（内存 < 10%）：
    卸载所有场景完整记忆 → 仅保留全局 + 概要
    停止接受新子Agent 任务
    日志级别降为 warn

存活模式（OOM 即将发生）：
    仅保留主Agent 对话 + 看门狗
    所有其他功能暂停
    持续写入 state.json 备份
```

### 6.4 自动恢复

```
系统资源恢复 → 检测到内存可用 > 30%
    ↓
阶段一：恢复内存索引 → 重新加载场景记忆
    ↓
阶段二：恢复浏览器 Context Pool → 接受搜索请求
    ↓
阶段三：恢复全量日志 → 正常模式
    ↓
通知用户："系统已恢复正常"
```

### 6.5 跨版本恢复

```
Bugs v1 崩溃 → 看门狗检测到新版本 v2 可用
    ↓
不立即回退到 v1
    ↓
如果 v1 连续崩溃 3 次 → 尝试 v2
    ↓
如果 v2 也崩溃 → 降级模式 → 通知用户手动处理
```

### 6.6 影子内存——双状态热恢复

**不增加第二进程。** 在已单一进程中维护两份内存副本，成本 ~10KB。

```
┌─────────────────────────────────┐
│         Bugs 进程                │
│                                  │
│  ┌──────────────┐               │
│  │  主状态        │  活跃读写       │
│  │  primary       │               │
│  │  (Arc<RwLock>) │               │
│  └───────┬──────┘               │
│          │ 后台 task             │
│          │ 每 100ms 同步          │
│          ▼                      │
│  ┌──────────────┐               │
│  │  影子状态      │  只读备用       │
│  │  shadow       │               │
│  └──────────────┘               │
│                                  │
│  primary 崩（panic 被 catch）     │
│      ↓                          │
│  shadow 立即升级为 primary       │
│  ~100ms 恢复（自上次同步的延迟）   │
└─────────────────────────────────┘
```

**同步的内容（仅关键状态，~10KB）：**

```rust
struct CriticalState {
    current_scene: SceneId,           // 当前场景
    active_subagents: Vec<SubAgentId>,// 活跃子Agent 列表
    pending_tasks: Vec<TaskId>,       // 未完成任务
    last_chat_context: String,        // 最后对话上下文
    permission_grants: Vec<Grant>,    // 本次会话权限授予记录
}
```

**恢复流程：**

```
主线程 panic
    ↓ catch_unwind 捕获
    ↓
① shadow → primary（影子升级，~0ms）
② 主线程用新 primary 重建 Agent 上下文
③ 通知用户："刚恢复，我们已经从上次中断的地方继续"
④ 异步：从磁盘加载非关键状态（全部场景记忆等）
⑤ 异步：从 state.json 补充遗漏
    ↓
~100ms 内用户无感恢复
```

**为什么不担心同步延迟：** 100ms 的同步间隔意味着最多丢失 100ms 的状态变更。这个窗口内用户最可能刚发完一句话——shadow 里已经同步了用户输入，只在等 Agent 响应。Agent 响应本身就需要数百 ms，100ms 的同步延迟在感知层面是不可见的。
