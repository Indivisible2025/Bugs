# 数据持久化设计

---

## 一、存什么

| 数据类型 | 大小估算 | 存储需求 |
|:--------|:-------:|:--------|
| 全局记忆 | ~1 MB | 持久化 |
| 场景概要 | ~100 KB/场景 | 持久化 |
| 完整场景记忆 | ~50 MB/场景 | 持久化 + 内存按需加载 |
| 配置文件 | ~10 KB | 持久化 |
| 会话历史 | ~1 MB | 持久化（可配置关闭） |
| 运行时状态 | ~100 KB | 仅内存，不持久化 |

## 三、怎么存——存储引擎

### 为什么选嵌入式 KV 引擎

| 对比 | PostgreSQL | SQLite | Sled (Rust) / BoltDB (Go) |
|:----|:---------:|:------:|:--------------------------:|
| 外部依赖 | ✅ 需要服务 | ❌ 文件 | ❌ 文件 |
| 内存占用 | ~50MB | ~1MB | ~100KB |
| 启动开销 | ~1s | ~50ms | ~1ms |
| 向量支持 | ✅ pgvector | ❌ 无 | 单独存储 |
| 二进制部署 | ❌ 需要外部 | ✅ | ✅ |

### 选定：redb（替代 Sled）

> ⚠️ Sled 从未发布稳定版本，作者坦言"也许永远不会稳定"。选择 redb 作为替代。

| 参数 | Sled | redb |
|:----|:----|:----|
| 稳定性 | ❌ 无稳定版 | ✅ 活跃维护，遵循 semver |
| 内存占用 | ~100KB | ~200KB |
| API 风格 | 异步 | 同步（无需 tokio 运行时） |
| Rust 生态 | 知名 | 增长中 |

备选方案：如果 redb 不满足需求，可回退到 SQLite（略重但对 Rust 有 rusqlite 成熟绑定）。

### 压缩引擎

所有持久化压缩默认使用 **zstd**：

| 场景 | 压缩方式 | 说明 |
|:----|:--------|:----|
| 记忆归档 | zstd level 3 | 场景退出时压缩完整记忆，zstd 比 zlib 省 30% 空间 |
| 备份导出 | zstd | `bugs memory export` 输出 .tar.zst |
| 会话历史 | zstd | 压缩 JSON 文件，level 1（快速） |
| 向量索引 | 不压缩 | 随机访问不适合压缩 |

### 向量引擎

| 决策 | 选择 | 理由 |
|:----|:---:|:----|
| 向量引擎 | hnsw_rs (Rust) | 纯 Rust，无外部依赖 |
| 默认维度 | 768 | BGE-base-en 输出维度 |
| 可切换模型 | ✅ | 每个场景可独立配置 embedding 模型 |
引擎：嵌入式 KV 引擎（随语言而定）
键格式：{layer}:{scene_id}:{key_hash}
值格式：序列化的 MemoryEntry（protobuf/msgpack）
向量索引：单独的文件，使用 hnsw 目录
```

**为什么不用 SQLite？**
- 不需要 SQL 查询——所有查询都是键值匹配或向量搜索
- SQLite 的 WAL 机制比纯 KV 多一层开销
- KV 引擎更轻，更符合 <500KB 目标

### 向量索引

```
向量引擎：hnsw (Rust) 或单独量化索引
存储位置：~/.bugs/data/scenes/<scene>/vectors.hnsw
维度：取决于 embedding 模型的输出（默认 768 = BGE-small-en）
```

## 四、持久化策略

### 写入策略

```
写入频率（可配置）：
    同步模式：每次写入立即 sync（最安全，但慢，适合桌面）
    批处理模式：每 10s 或 1000 条写一次（默认，适合服务器）
    纯内存模式：进程退出前才写入（最快，容忍丢数据）
```

### 压缩策略

```
触发条件：
    - 完整记忆 > 50 MB（达到上限）
    - 手动触发 /memory compact
    
压缩步骤：
    按信任值排序 → 保留 top 80% → 剩余语义压缩为概要
    → 概要存入 summary.sled
    → 删除完整记忆中低信任条目
```

## 五、备份与恢复

```
导出：bugsctl memory export <scene> --output backup.tar.gz
导入：bugsctl memory import backup.tar.gz
自动备份：可配置定时导出（默认关闭，用户自己管）
```

## 七、升级与迁移

### 升级策略

```
bugs upgrade                    → 检查新版本 → 下载 → 替换二进制 → 重启
bugs upgrade --dry-run          → 只检查，不安装
bugs upgrade --version x.y.z    → 指定版本
```

### 数据格式迁移

```
启动时检测数据格式版本号
    ↓
当前格式版本 > 存储格式版本 → 自动迁移：
    ├── 小版本升级（0.1.0 → 0.1.1）→ 透明迁移，无需干预
    ├── 大版本升级（0.x → 1.0）     → 提示用户备份 → 执行迁移
    └── 无法自动迁移                → 提示手动迁移或回退
```

### 迁移保障

| 保证 | 说明 |
|:----|:----|
| 迁移前自动备份 | 旧数据导出到 ~/.bugs/backups/YYYY-MM-DD/ |
| 迁移失败可回退 | 降级旧版本后数据立即可用 |
| 跨机器迁移 | `bugs memory export --all` → `bugs memory import` |

## 八、假性能自检

| 调度引擎犯的错 | 持久化有没有？ | 说明 |
|:-------------|:----------|:----|
| 固定权重 | ❌ | 无评分 |
| 两阶段截断 | ⚠️ 压缩保留 80% | 不是截断，是语义压缩保留，低信任条目被清除是合理行为 |
| 立即重试 | ❌ | 写入失败直接报错，不自动重试（丢数据风险） |
| 分段跳跃 | ❌ | 压缩阈值 50MB 是硬上限，越过立即触发 |
| 无上限 | ❌ | 每场景 50MB 硬上限，全局 1GB 上限 |
